# Правила и условия для агентов — Polymarket Analytics Agent

## Обязательные правила

### R1. Формула P&L
При расчёте P&L для ставок Polymarket:
- Использовать **единую** формулу для YES и NO: `pnl = bet * (1/price - 1)` при выигрыше, `pnl = -bet` при проигрыше.
- Переменная `price` — цена **купленной** доли (в Excel — колонка 7). Для YES это yes_price, для NO — no_price.
- **Запрещено** для NO использовать: `1/(1-price)`, `no_price = 1 - price`, или любые производные от YES-цены.

### R2. Сторона ставки (YES/NO)
При определении выигрыша обязательно учитывать сторону:
- `won = (outcome == "Yes")` только если ставка была YES.
- `won = (outcome == "No")` только если ставка была NO.
- Скрипты `update_results`, `check_and_update` и т.п. должны читать колонку 4 (Направление) и использовать её в логике.

### R3. Floating P&L для активных ставок
Для оценки текущей стоимости позиции:
- **YES:** `value = (bet/price) * cur_yes_price`
- **NO:** `value = (bet/price) * cur_no_price`
- **Запрещено** использовать `cur_yes_price` для оценки NO-позиции.

### R4. MANUAL_RESULTS (refresh.py)
- Ключ словаря — **точная** строка из колонки 3 Excel (вопрос рынка).
- `outcome`: `"Yes"` | `"No"` (с заглавной буквы) или `None` (пропуск, заполнить позже).
- При `outcome is None` запись не обрабатывается (проверка: `if manual and manual.get("outcome")`).

### R5. Не придумывать данные
- Не вносить результаты матчей в MANUAL_RESULTS без подтверждённого источника.
- При отсутствии данных писать «нет данных», не симулировать.
- При 403 от API использовать кэш (agent-tools), не подставлять фейковые ответы.
- **Сообщать о недостаточности данных:** если данных мало или они неполные — явно указывать это, не выдавать частичные выводы за полноценные.
- **Сообщать о надёжности данных:** при цитировании статистики, P&L, ROI, WR — указывать источник и степень доверия (опубликовано в репо, из сторонних статей, агрегатная оценка, нет публичных данных).

## Рекомендуемые условия

### C1. Перед изменением формул
- Прочитать `FORMULAS_AUDIT.md`.
- Прогнать тест: NO, bet=30, price=0.27, won=True → pnl=81.11.

### C2. Перед добавлением новых скриптов обновления
- Проверить, не дублирует ли логику `refresh.py` или `check_and_update.py`.
- Использовать ту же формулу P&L, что и в refresh.py.

### C3. Пути и расположение
- Excel: по умолчанию `polymarket_рекомендации.xlsx` в корне репо; переопределение — `REFRESH_EXCEL_PATH` в `.env` (см. `docs/RUNBOOK.md`).
- Кэш рынков для `refresh.py`: `REFRESH_MARKET_CACHE_FILES` / `REFRESH_MARKET_CACHE_DIR` или `data/market_cache/`.
- План (копия/ссылка): `M5Unified\src\utility\.cursor\plans\polymarket_analytics_agent_b5985182.plan.md`

### C4. Фильтры ставок (из плана)
- NO не ставить, если yes_price > 0.65.
- Использовать whitelist лиг (Tier1, Tier2, SKIP).
- Ликвидность мин. 2000–3000$.

### C5. После правок
- Запустить `python refresh.py`.
- Проверить, что P&L закрытых ставок сходится с ожидаемым (скрипт верификации из контекста).

## Структура Excel (справочно)

| Колонка | Название        | Смысл                          |
|---------|-----------------|--------------------------------|
| 2       | Condition ID    | ID рынка (короткий)            |
| 3       | Рынок (вопрос)  | Текст вопроса, ключ для MANUAL_RESULTS |
| 4       | Направление     | YES, NO или ДА                 |
| 6       | Ставка $        | Сумма ставки                   |
| 7       | Цена доли       | Цена купленной доли (YES или NO) |
| 8       | Потенц. выигрыш | bet * (1/price - 1)            |
| 10      | Результат       | ВЫИГРЫШ / ПРОИГРЫШ / Ожидание   |
| 11      | P&L $           | Фактический P&L                |
