# Три основные задачи и их реализация

## 1. Три основные задачи

---

### Задача 1: fix-draw-probability-correction (коррекция Kelly на ничью)

**Значение для проекта**

2 из 6 проигрышей — ничьи (Aston Villa 1:1, Tolima 1:1). В бинарных рынках Polymarket «Will X win?» ничья всегда = NO. Рынок оценивает P(YES) ≈ вероятность победы, но Kelly работает с P(исход YES). В футболе ничья — 25–30% исходов, и без поправки Kelly завышает ставки.

**Влияние:** снижение риска на футбольных ставках на победителя, более реалистичный размер ставки.

---

### Задача 2: fix-team-form-analysis (анализ формы команды)

**Значение для проекта**

Ставки на андердогов (Tolima 0.295, Tijuana 0.265) делались без оценки формы и состава. Отсутствует проверка: в какой форме команда, как играет дома/в гостях, есть ли травмы. Без этого оценка edge ненадёжна.

**Влияние:** фильтр слабых сигналов, меньше ставок на команды в плохой форме.

---

### Задача 3: fix-forward-test-volume (50+ сигналов в forward-test)

**Значение для проекта**

При 8 ставках 95% доверительный интервал для WR — [3%, 65%]. Нельзя понять, убыток — из-за стратегии или случайности. Нужна база сигналов (включая «виртуальные» — не исполненные) для калибровки стратегии.

**Влияние:** обоснованное решение о масштабировании ставок или изменении правил.

---

## 2. Решение реализации и возможные исходы

---

### Задача 1: draw-probability-correction

**Реализация**

1. В `config.py` — словарь `DRAW_PROB_BY_LEAGUE`:
   - `epl: 0.27`, `bundesliga: 0.26`, `liga_mx: 0.30`, `colombia: 0.30`, `nba: 0.0`, `ufc: 0.0`.

2. В `analysis/kelly.py` — параметр `draw_prob: float = 0.0` в `compute_kelly` и вызовах из `signals.py`.

3. Логика: для рынка «Will X win?» при `draw_prob > 0`:
   - `p_win_adjusted = true_prob / (1 - draw_prob)` (Pinnacle даёт 3-way odds: win/draw/lose).
   - Вариант: если Pinnacle только h2h — `p_yes_from_market` скорректировать на ничью: `p_win_implied = pm_yes / (1 - draw_prob)` для сравнения с Pinnacle.

4. В `signals.py` — определять лигу по `odds_line.sport_key` или `market.question`, передавать `draw_prob` в `compute_kelly`.

**Возможные исходы**

| Исход | Вероятность | Влияние на проект |
|-------|-------------|-------------------|
| Kelly уменьшается на 15–25% для футбола | Высокая | Меньше ставок и объём на match-winner, меньше потерь от ничьих |
| Ставки на футбольные андердоги падают до 0 | Средняя | Часть сигналов теряется, но риск снижается |
| Переоценка draw_prob | Низкая | Нужна тонкая настройка по лигам после накопления данных |

---

### Задача 2: team-form-analysis

**Реализация**

1. Новый модуль `data/team_stats.py`:
   - API: Football-data.org (бесплатно, ~10 req/min).
   - Функции: `get_team_form(team_name, league, last_n=5)` → `{wins, draws, losses, goals_for, goals_against}`.
   - Вспомогательно: `get_team_id_by_name()` для маппинга Polymarket/Pinnacle → Football-data.

2. В `analysis/signals.py` — пост-фильтр `TeamFormFilter`:
   - Для каждого сигнала с матчем (есть home/away из OddsLine) — запрос формы.
   - Правило: не рекомендовать YES на победу, если команда выиграла < 2 из последних 5.
   - Опционально: пропускать, если форма < 40% (2W из 5).

3. Обработка ошибок: при недоступности API — пропускать проверку, не блокировать сигнал.

4. Кэш: Redis или простой JSON-кеш на 1–2 часа, чтобы не дергать API на каждый цикл.

**Возможные исходы**

| Исход | Вероятность | Влияние на проект |
|-------|-------------|-------------------|
| Сокращение 20–40% футбольных сигналов на андердогов | Высокая | Меньше ставок на слабые команды, более осознанный риск |
| API-лимиты или неполные данные | Средняя | Часть матчей без проверки формы; нужен fallback |
| False negative: отсев выигрышных сигналов | Средняя | Небольшое снижение числа ставок; приоритет — защита от слабых команд |
| Новый источник edge (форма) | Низкая | Дополнительная валидация сверх Pinnacle |

---

### Задача 3: forward-test-volume

**Реализация**

1. Расширение `storage/db.py`:
   - Использовать таблицу `snapshots` для всех сигналов с edge > 0, а не только исполненных.
   - В `executor.py` — при пропуске сигнала (risk, slippage, время) тоже писать snapshot с `recommended_bet_usd`, `pm_price`, `pinnacle_true_prob`, `edge`.
   - Добавить колонки: `league`, `side`, `resolved`, `outcome_won`, `virtual_pnl` (если уже нет).

2. В `analysis/backtest.py`:
   - `show_signal_calibration()`: по resolved snapshots — `predicted_prob vs actual_won`, калибровочный график/таблица.
   - Группировка: по лиге, side (YES/NO), диапазону edge (0.03–0.05, 0.05–0.08, >0.08).
   - Метрики: реальный WR, ожидаемый WR из edge, ROI по группам.

3. Цель: ≥ 50 записей в snapshots (желательно исполненных или «виртуально» исполненных по логике стратегии).

4. CLI: `python main.py --stats` или отдельная команда для вывода калибровки.

**Возможные исходы**

| Исход | Вероятность | Влияние на проект |
|-------|-------------|-------------------|
| Реальный WR близок к ожидаемому | Средняя | Подтверждение стратегии, можно повышать объём ставок |
| Реальный WR ниже ожидаемого | Средняя | Пересмотр min_edge, Kelly, фильтров лиг |
| Сильная разница по лигам | Высокая | Ужесточение или отключение слабых лиг |
| Недостаточно данных (< 50) | Высокая (сейчас) | Продолжать копить snapshots 2–4 недели |
| Калибровка выявляет системный bias | Низкая | Доработка формул или источников p_true |

---

## Сводная таблица исходов для проекта

| Задача | Лучший исход | Нейтральный | Худший исход |
|--------|--------------|-------------|--------------|
| Draw correction | Меньше убытков от ничьих, более устойчивый P&L | Небольшое снижение ставок | Слишком консервативно, мало сигналов |
| Team form | Меньше ставок на слабые команды, выше WR | Умеренное уменьшение сигналов | API нестабилен, много false negative |
| Forward-test | Подтверждение стратегии, уверенное масштабирование | Нужно больше данных | Выявлены дефекты стратегии, требуется переработка |

---

## Рекомендуемый порядок

1. **Draw correction** — быстрая реализация, явный эффект на футбол.
2. **Forward-test volume** — запустить сбор данных параллельно.
3. **Team form** — подключить после стабилизации двух первых пунктов.
